# FeaturePilot 四命令流程验证与修复设计

**日期：** 2026-07-12
**插件仓库：** `D:\01-code\feature-pilot`（保持当前 `dev/1.0/zrf`）
**测试项目：** `D:\02-canway\01-code\cw-auto-ops`（创建并切换 `dev/1.0/ai`）

## 1. 目标

在真实 Claude Code 插件会话中依次运行 `fp-init`、`fp-prd`、`fp-start` 和 `fp-quick`，验证 FeaturePilot 的交互流程、阶段门禁、产物路径、流程合理性和代码生成行为。发现插件问题时，先保存可重复证据和定位根因，再在 `feature-pilot` 中补最小回归检查并修复。

本轮允许插件在 `cw-auto-ops` 中生成文档和最小业务代码，但不运行该项目的单测、构建、服务或浏览器验收。所有测试产物和业务代码改动保留在 `cw-auto-ops` 的 `dev/1.0/ai` 分支供人工复查。

## 2. 基线与边界

- 先运行 `feature-pilot/scripts/validate-plugin.ps1`，记录插件静态契约基线。
- 使用当前插件源码直接加载插件，避免已安装缓存版本干扰。
- `cw-auto-ops` 当前工作树必须干净；从当前提交创建 `dev/1.0/ai`。
- 不切换 `feature-pilot` 分支，不提交或推送测试项目改动。
- 允许修改 `feature-pilot` 以修复已确认缺陷；不做与四个命令无关的重构。
- 不读取或复制 secrets，不安装依赖，不运行目标项目的业务验证命令。
- 由于不执行目标项目测试，本轮只判断“插件流程是否走通”和“生成行为是否符合契约”，不声明生成的业务代码功能正确。

## 3. 测试方法

采用真实 Claude Code 多轮会话为主、插件静态校验为辅的方法：

1. 通过 `claude --plugin-dir D:/01-code/feature-pilot` 在 `cw-auto-ops` 中加载当前插件。
2. 每条流程使用可恢复会话，逐轮提供确认和选择，验证门禁能停住并正确恢复。
3. 每轮记录：用户输入、助手响应、需要的交互轮次、文件变更、阶段状态和异常。
4. 在关键确认前后检查 `git status` 与目标路径，证明是否存在提前写入。
5. 对生成的 FeaturePilot 产物运行结构检查；目标项目业务测试不运行。
6. 如发现缺陷，遵循“复现 → 根因 → 最小失败检查 → 单一修复 → 重跑相关流程”的顺序。

若非交互模式不能可靠恢复插件会话，则使用交互式 Claude Code 子进程完成同样的逐轮步骤；不得退化为仅阅读 Skill 文件并声称 E2E 通过。

## 4. 流程矩阵

### 4.1 `fp-init`

验证：

- 在项目根创建 `fp-docs/manifest.md`、`settings/`、`intel/`，不创建 `changes/`、`archive/`、`history/`。
- 正确识别 Canway/CW 项目信号，并在采用项目族示例前取得明确同意。
- 可选 settings 逐项说明用途，允许跳过；轻量 intel 只读发现且 Unknown 不猜测。
- 首次完成后再次调用，验证现有文件不会未经批准被覆盖。
- 最终报告路径、创建/跳过项、关键 Unknown 和下一步命令。

### 4.2 `fp-prd`

选择一个与 `cw-auto-ops` 现有实现相关、范围有限且可继续开发的需求。验证：

- 先运行共享只读探索，只把代码事实用于减少技术不确定性，不替用户决定产品需求。
- Bucket A/B 批量复核；Bucket C 每轮只问一个问题，默认目标 3–5 个。
- 最终确认摘要明确列出需求、假设、开放问题、原型决定和 canonical form。
- 摘要获批前不创建 change 目录或 PRD。
- 获批后只生成 `prd.md` 或 `prd/00-index.md` 之一，并符合强制模板和大小限制。
- 最终输出包含可复制的 `/fp-start <slug>`。

### 4.3 `fp-start`

使用上一阶段生成的 slug。验证：

- 复用 resolved PRD，不重复完整需求访谈。
- 依次执行 proposal、design、plan，每个阶段写入并核验产物后等待明确确认。
- 设计阶段只由一次 `fp-brainstorm` 上下文完成，不启动第二个设计收尾流程。
- 计划确认前不修改业务代码。
- 执行前显式展示并等待 Direct/SDD 选择；不得自动替用户选择。
- 选择适合最小测试需求的执行方式，完成代码生成和最终 review 流程。
- 不运行目标项目单测、构建或服务；如子流程默认要求验证，则明确告知本轮测试范围并将其记录为未执行风险，不伪造通过结果。

### 4.4 `fp-quick`

选择一个与完整链路需求不同的独立小改动。验证：

- 使用共享 `fp-explore quick` 聚焦候选文件、复用模式和验证路径。
- 无阻塞问题时直接给内联计划；有阻塞问题时每轮至多一个实质性问题。
- 计划确认前不修改业务代码。
- 不创建 `fp-docs/changes/` 下的新 change 文档。
- 确认后生成最小代码改动并报告应执行但本轮未执行的验证。

## 5. 问题分类

### P0/P1：流程正确性缺陷

包括命令或 Skill 未加载、错误路由、错误路径、确认前写入、跳过阶段、非法 canonical 产物、无法恢复、未显式选择执行模式、`fp-quick` 越界生成完整文档链等。此类问题必须修复。

### P2：流程合理性或繁琐度问题

包括重复读取、重复询问已确认事实、与风险不匹配的额外门禁、无关扫描、阶段摘要无法复用和不必要的长解释。仅在不削弱需求确认、执行确认、覆盖保护和高风险选择门禁的前提下优化。

### P3：输出质量问题

包括下一步不清楚、路径/状态报告不完整、推荐缺少理由和产物重复难审阅。与四个目标命令直接相关时修复，否则记录为后续项。

## 6. 根因与修复协议

每个问题必须记录：

1. 精确命令、会话轮次和前置工作树状态。
2. 实际输出和实际文件变化。
3. 对应 Skill/command 契约中的期望行为。
4. 从命令适配器、Skill、共享 workspace/artifact contract 到文件写入点的数据流。
5. 单一根因假设及最小验证。
6. 能在修复前失败的自动化检查；若真实模型行为无法稳定断言，则使用最接近根因的契约/fixture 检查，并保留真实会话复现记录。
7. 仅针对根因的最小修复。
8. 相关命令流程重跑和完整插件静态校验结果。

同一问题连续三次修复失败时停止叠加补丁，重新评估流程架构并向用户报告。

## 7. 验收标准

- `cw-auto-ops` 位于 `dev/1.0/ai`，测试产物和代码生成改动保留。
- `fp-init` 首次初始化和再次调用的覆盖安全流程均走通。
- `fp-prd` 完成真实多轮访谈并生成唯一合法 canonical PRD。
- `fp-start` 复用 PRD，完成 proposal、design、plan、显式执行选择、代码生成和 final review。
- `fp-quick` 完成独立小需求的探索、内联计划确认和代码生成，且无新增 change 文档。
- 所有已确认插件缺陷均有复现、根因、回归检查、修复和重跑证据。
- `scripts/validate-plugin.ps1` 在最终状态通过。
- 最终报告分开说明：已验证插件流程、未运行的目标项目测试、生成代码的未验证风险和保留文件。

## 8. 明确不在范围内

- 不验证 `cw-auto-ops` 业务功能是否正确。
- 不运行该项目的单测、lint、构建、数据库迁移、服务或浏览器测试。
- 不提交或推送任何仓库。
- 不测试 `fp-explore`、`fp-propose`、`fp-brainstorm` 等作为独立用户入口；它们仅作为四条目标流程的内部阶段被观察。
- 不修复与本次四命令流程无直接关系的历史问题。
