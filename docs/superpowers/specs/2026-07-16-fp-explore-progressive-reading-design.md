# FeaturePilot `fp-explore` 渐进式低上下文读取设计

**日期：** 2026-07-16
**状态：** 设计已确认并实施
**范围：** 所有非空 `fp-explore` 探索；`quick` profile 使用更严格的读取预算

## 1. 背景

当前 `fp-explore` 已要求“先搜索、再打开文件”，并通过 `tiny`、`small`、`standard`、`max` 约束文件读取、搜索和 Git 检查次数。但现有合同仍存在三个缺口：

1. “读取一个文件”没有区分局部范围读取与无范围整文件读取；
2. 搜索命中后没有明确要求使用 `offset`/`limit` 缩小上下文；
3. `quick` 虽然使用 `small` 预算，仍可能把多个候选源文件逐个整读，导致探索阶段延迟和上下文成本偏高。

因此，本次调整不改变 `fp-explore` 的只读边界、内部调用/返回 schema 或调用方职责，而是在现有预算之上加入一套可验证的渐进式读取策略。

## 2. 目标与非目标

### 2.1 目标

1. 所有非空探索默认采用 `Glob → Grep → 局部 Read → 必要时整读`。
2. 优先以路径、符号、命中行和局部上下文建立证据，避免无依据整读文件。
3. 给整文件读取设置明确升级条件，并要求在调用前说明具体理由。
4. 为 `quick` profile 设置更严格的候选数量、局部窗口和无范围读取预算。
5. 保持现有 `tiny`、`small`、`standard`、`max` 总预算和内部 contract 向后兼容。
6. 用静态合同测试防止策略退化。

### 2.2 非目标

- 不禁止所有整文件读取；
- 不修改 `fp-explore-invoke` 或 `fp-explore-return` 字段；
- 不新增运行时解析器、工具代理层或自动改写 `Read` 调用的 hook；
- 不改变 `fp-prd`、`fp-start`、`fp-quick` 的产品决策、确认、计划、实现或验证职责；
- 不引入子 Agent 作为单一搜索问题的默认路径；
- 不改变外部研究授权、敏感数据或只读安全边界。

## 3. 总体方案

在 `skills/fp-explore/SKILL.md` 的仓库调查流程与预算规则之间加入统一的 **Progressive low-context inspection** 合同。

该合同适用于：

- 非空 public standalone 探索；
- `prd-facts`；
- `start-routing`；
- `quick`。

空输入 standalone 继续执行现有有界项目概览，不扩大到业务源码树；它只需遵守“读取最小必要范围”的通用原则，不使用 `quick` 的专属数值预算。

## 4. 渐进式调查流水线

### 4.1 阶段 A：定位候选路径

1. 将目标转换为精确文件名、目录、符号、路由、API、模型、组件、配置项、错误文本和测试名称。
2. 优先使用 `Glob` 或等价的路径搜索定位候选文件。
3. 候选列表应按与目标的直接相关性排序，而不是把所有匹配路径都进入读取阶段。
4. 排除依赖、生成文件、缓存、构建输出、二进制、大型 fixture、历史归档和用户未授权的敏感范围。

路径搜索已经足以回答“文件是否存在、位于何处”时，不再打开文件以重复确认。

### 4.2 阶段 B：定位符号和命中行

1. 对候选文件或限定目录使用 `Grep` 或等价内容搜索。
2. 优先搜索精确符号、导入、调用点、路由、字段、组件名和测试名；仅在精确词无结果时扩大关键词。
3. 搜索应返回文件路径、行号和必要上下文。
4. 搜索结果已经足以支持事实时，直接引用命中证据，不为了“更全面”而打开文件。

禁止在一次广泛关键词搜索后，直接整读全部命中文件。

### 4.3 阶段 C：局部读取

1. 搜索命中后，默认用 `Read(offset, limit)` 或等价范围读取查看上下文。
2. 默认窗口为 **80–160 行**。
3. 窗口边界应尽量覆盖一个完整函数、类、组件区块、序列化器、路由组或测试用例，而不是机械截取固定行数。
4. 如果命中靠近文件开头或结尾，可缩短窗口；如果一个语义单元略超过 160 行，可读取相邻窗口，但必须保持范围聚焦。
5. 不重复读取已经覆盖的同一文件范围。
6. 每次读取都记录实际范围，供 `inspected-scope` 和预算状态使用。

### 4.4 阶段 D：按证据升级

局部读取不足时，按以下顺序升级：

1. 读取同一语义单元的相邻窗口；
2. 搜索缺失的定义、引用或测试；
3. 读取第二个直接相关的局部窗口；
4. 只有仍存在关键证据缺口时，考虑无范围整文件读取。

不得因为“这个文件看起来重要”直接跳过搜索和局部读取。

## 5. 整文件读取门槛

### 5.1 默认门槛

对于已知或通过只读静态检查确认 **超过 300 行**的应用源码、测试或配置文件，默认不得使用无 `offset`/`limit` 的 `Read`。

文件行数未知时，可以使用状态中立的行数/结构检查；该操作计入现有 `Searches/static inspections` 预算。不得先整读文件再以“原来超过 300 行”为理由追认。

### 5.2 允许升级的情况

满足以下至少一个条件时，可整读文件：

1. 文件不超过 300 行，且整体结构与目标直接相关；
2. 控制流或数据流分散在多个非相邻区域，局部窗口无法可靠还原；
3. 模块注册、导入副作用、生命周期、路由聚合或配置覆盖顺序需要全局上下文；
4. 单文件组件的 template、script、状态和方法高度耦合，局部读取会丢失关键交互关系；
5. 已读取两个直接相关局部窗口，仍存在相互矛盾的模式或影响结论的关键缺口；
6. 用户明确要求全面审查该文件，且仍符合安全与预算边界。

### 5.3 升级说明

执行无范围整文件读取前，必须给出一句具体原因，说明：

- 当前还缺少什么证据；
- 为什么搜索或局部窗口不能解决；
- 为什么该文件的全局结构与目标直接相关。

“需要更多上下文”“该文件很重要”“为了完整理解”不构成充分理由。

## 6. 预算模型

### 6.1 保留现有总预算

现有预算表保持不变：

| Budget | File reads | Searches/static inspections | Local Git inspections | External sources |
|---|---:|---:|---:|---:|
| `tiny` | 6 | 4 | 1 | 0 unless approved |
| `small` | 12 | 8 | 2 | 0 unless approved |
| `standard` | 24 | 14 | 3 | 0 unless approved |
| `max` | 40 | 20 | 5 | 0 unless approved |

这些计数继续约束探索总量；新增计数是叠加约束，不替代现有预算。

### 6.2 新增读取计数

每次非空探索同时跟踪：

- `candidate paths`：进入进一步内容调查的候选路径数量；
- `local read windows`：使用明确范围读取的窗口数量；
- `unbounded application-file reads`：对应用源码、测试或配置执行的无范围读取数量。

计数规则：

- 同一文件两个不同范围算两个局部窗口；
- 重叠且没有新增证据的重复范围禁止读取；
- 一个文件先局部读取、后整读，分别计入局部窗口和无范围读取；
- root instructions、短 manifest/index 和明确短小的 FeaturePilot contract 不计入“应用文件无范围读取”专属计数，但仍计入现有文件读取预算，并继续遵守最小必要读取原则；
- 范围读取仍使该文件计入现有“distinct file content read”预算，避免绕过总文件数限制。

### 6.3 `quick` 专属严格预算

每次 `quick` profile 调用使用以下上限：

| 计数器 | 上限 |
|---|---:|
| 初始候选路径 | 8 |
| 局部读取窗口 | 8 |
| 应用文件无范围读取 | 1 |

规则：

1. 候选超过 8 个时，按直接相关性保留最强证据路径，并把其余路径列为未检查范围。
2. 达到 8 个局部窗口后，不得通过继续扩大搜索来隐式突破 quick 的定位目标。
3. 唯一一次整读必须满足第 5 节升级条件；它不是默认配额。
4. 预算达到上限时：
   - 证据已足够则返回 `complete`；
   - 仅部分范围可支持则返回 `partial`；
   - 无法可靠判断则返回 `blocked`。
5. 返回中列出达到的具体计数器和未检查区域，由 `fp-quick` 决定缩小问题、重新调用或建议升级到 `fp-start`。

### 6.4 其他模式/profile

- `prd-facts`、`start-routing` 和非空 standalone 使用同一渐进顺序与整读门槛；
- 继续受其当前预算 profile 约束；
- 没有 `quick` 的固定 `8 / 8 / 1` 上限，但必须停止在“足够证据”，不得把总预算当作必须用完的配额；
- `max` 仍要求具体跨模块理由。

## 7. 并行与子 Agent

1. 多个相互独立且候选范围已知的 `Glob`/`Grep`/局部 `Read` 可以并行执行。
2. 不得并行发起多个未知范围的整文件读取。
3. public standalone 遇到真正独立的跨模块调查时可以建议或使用只读探索子 Agent，但每个子 Agent 必须遵守相同渐进读取合同，并只返回摘要、路径、行号和未检查范围。
4. `quick` 不为单一搜索问题启动子 Agent；只有目标已经明显分裂为多个独立工作流时，才将其作为“不适合 quick”的范围证据，而不是通过并行扩大 quick。

## 8. 返回与可观察性

不修改 `fp-explore-return` schema。现有字段按以下方式承载新增信息：

- `inspected-scope`：包含搜索范围、读取文件和局部行号范围；
- `budget-status`：预算耗尽时使用现有 `budget-exhausted:<counter>` 形式，例如 `budget-exhausted:quick-local-read-windows`；
- `verified-facts`：继续提供 `path:line` 证据；
- `risks`：记录因为未整读而无法排除的真实风险；
- `next-caller-action`：说明应缩小目标、补充决策、再次探索或由 caller 升级流程。

public standalone 的进度文本保持简洁。仅在发生整读升级、改变调查方向或遇到预算/证据阻塞时说明；不逐条叙述普通搜索和局部读取。

## 9. 错误与边界处理

1. `Glob` 无结果：检查目标词和用户指定范围，不立即扫描整个仓库。
2. `Grep` 无结果：最多进行一次有依据的同义词、命名风格或路径扩展，再决定是否返回未知。
3. 局部窗口边界截断语义单元：读取一个相邻窗口，不直接整读文件。
4. 候选模式互相冲突：优先搜索调用方和测试；仍无法判断时返回风险或 caller-owned question。
5. 工具不支持范围读取：使用可用的专用搜索/结构工具；如果只能整读且违反门槛，则报告限制，不以 Bash 转储整个文件绕过合同。
6. 敏感文件、外部研究、写操作和可能产生副作用的命令继续遵守现有禁止/授权边界。

## 11. 测试与验证

### 11.1 静态合同断言

`scripts/test-explore-contract.ps1` 必须验证：

1. 顺序明确包含 `Glob → Grep → local/ranged Read → justified full-file Read`；
2. 默认局部窗口为 `80–160` 行；
3. 大于 300 行的应用文件默认禁止无范围读取；
4. 整读升级至少列出可验证条件，并要求具体理由；
5. `quick` 包含 8 个候选、8 个局部窗口、1 个整读上限；
6. 达到 quick 上限后返回 complete/partial/blocked，而不是静默继续；
7. 现有总预算表和内部 schema 未改变；
8. public command 仍是薄适配器；
9. 只读、安全、研究授权和 caller ownership 没有减弱。

### 11.2 防退化负例

验证器应拒绝与以下含义等价的合同文本：

- 搜索后整读所有命中文件；
- 大文件可以为了完整理解直接整读；
- quick 可按需突破窗口或整读上限；
- 局部读取不计预算；
- 通过 Bash 输出整个文件绕过 `Read` 门槛；
- 单一 quick 搜索默认启动多个子 Agent 扩大调查。

### 11.3 执行验证

实施后运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\test-explore-contract.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\validate-plugin.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\measure-context.ps1
git diff --check
```

所有命令必须报告实际结果。失败时不得宣称更新完成。

## 12. 完成标准

仅当以下条件全部满足时，本次更新完成：

- 所有非空探索遵循渐进式低上下文调查；
- `quick` 的 `8 / 8 / 1` 限制可由静态合同验证；
- 大文件整读需要明确证据和可观察理由；
- 现有内部调用/返回合同及 caller gates 保持兼容；
- focused contract、完整插件验证、上下文测量和 diff 检查全部通过；
- 没有引入运行时组件、额外工作流状态或无关改动。
